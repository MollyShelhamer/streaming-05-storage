## Custom Project

### Dataset

- The dataset file is `data/sales.csv`.
- It contains sale records for orders, customers, regions, products, and
  payment metadata.
- Each record includes fields such as `order_id`, `datetime`, `region_id`,
  `currency_code`, `product_id`, `unit_price`, `quantity`, `is_online`,
  `customer_id`, and `payment_method`.
- The original sales dataset was used, not a modified version.

### Kafka Messages

- The producer sends full sale records as Kafka messages.
- The messages are published to the configured sales topic.
- No custom message key is introduced in this example.
- The message fields remain the same as the source sales records.

### Consumer Processing

- The consumer receives records from Kafka and validates required fields.
- It consumes up to the configured maximum messages and stops on timeout.
- It logs processing details, including subtotal, tax, total, and running
  statistics.
- It writes accepted records to `data/output/consumed_sales.csv`.
- It enriches each record with derived fields and appends a new
  `processed_timestamp` field before writing.

### Experiments

- Phase 4 change: added a new output CSV field,
  `processed_timestamp`, to track when each record was stored.
- Phase 5 application: wrapped `write_valid_record(OUTPUT_DB, enriched)` in a
  `try/except` so a DuckDB write failure is logged and the consumer continues.

### Results

- The producer and consumer ran successfully with the modified consumer
  logic.
- Accepted records were enriched, written to DuckDB, and appended to the CSV.
- On a DB write error, the consumer logs the failure and keeps processing.

### Interpretation

- This workflow shows how Kafka can carry sales records through a validation
  and enrichment pipeline.
- The edit changed the consumer from a strict failure mode to a more resilient
  stream processor.
- Watching messages move through Kafka demonstrates the value of a
  continuous data pipeline for business analytics.
- The consumed stream provides business intelligence on sales totals,
  running averages, and the timing of processed orders.
